# Clustering Paper Outputs

## 1. 本目录是什么

本目录保存的是**论文正式口径**下的聚类与类型比较结果。  
这版聚类不再直接沿用旧实验口径，而是建立在当前论文版 IPA 指标之上。

对应脚本：

- `旅游景点分析/scripts/08_run_experiment8_paper_clustering.py`

上游输入：

- `旅游景点分析/outputs/ipa_paper/experiment7_theme_panel.csv`
- `旅游景点分析/data/sentiment/experiment5_snownlp_scored.csv`

## 2. 为什么这样做聚类

本轮聚类的唯一目标是：**尽量强的可解释性**。

因此我们没有把“景区原有类别”直接放进聚类模型，而是采用三层结构：

1. `景区原有类别比较`
2. `治理画像聚类`
3. `治理紧迫度分层`

这样每一层都回答不同问题：

- 景区原有类别比较：回答“哪些类型景区更容易出现什么治理问题”
- 治理画像聚类：回答“哪些景区在问题结构上相似”
- 治理紧迫度分层：回答“哪些景区更需要优先治理”

## 3. 为什么不把景区原有类别直接作为聚类输入

原因有 3 个：

1. 景区原有类别是先验分类变量，适合做解释，不适合做无监督聚类输入。
2. 如果把它编码进模型，聚类容易变成“复现原有类别”，而不是发现治理结构。
3. 当前样本只有 20 个景区，各类别数量不平衡，直接混入类别变量会降低聚类稳定性。

因此：

- `category_name` 用于聚类后的解释与对照
- 不用于聚类计算本身

## 4. 主聚类的特征怎么来

聚类特征全部来自论文版 IPA 的主题层结果。

当前使用的一级治理主题是：

- `票务预约与入园体验`
- `现场服务与导览体验`
- `交通与开放空间体验`

对每个景区，构造以下特征：

### 4.1 主题优先级特征

- `priority_index__票务预约与入园体验`
- `priority_index__现场服务与导览体验`
- `priority_index__交通与开放空间体验`

### 4.2 主题不满意特征

- `dissatisfaction__票务预约与入园体验`
- `dissatisfaction__现场服务与导览体验`
- `dissatisfaction__交通与开放空间体验`

### 4.3 综合特征

- `total_priority_index`
  3 个治理主题 `priority_index` 的总和
- `mean_dissatisfaction`
  3 个治理主题 `dissatisfaction` 的平均值

## 5. 聚类方法

### 5.1 标准化

聚类前对所有数值特征做标准化：

- 方法：`StandardScaler`

原因：

- 不同特征量纲不同
- 不标准化会导致大尺度变量主导距离计算

### 5.2 聚类算法

主聚类使用：

- `AgglomerativeClustering`
- linkage：`ward`

选择理由：

- 样本量只有 20 个景区，层次聚类更适合小样本
- `Ward` 连接方式有利于得到类内更紧凑、解释更稳定的结果

### 5.3 K值比较

比较范围：

- `k = 2, 3, 4, 5`

原因：

- 样本量小，不适合把 `k` 设得过高
- 太少的类会丢失治理差异
- 太多的类会造成解释冗余

### 5.4 选K标准

使用：

- `silhouette_score`

即轮廓系数。

本轮结果中，轮廓系数最高的是：

- `k = 4`

因此主结果采用 4 类方案。

## 6. 为什么正文建议用“宏观三类”

虽然数据上最优的是 `k = 4`，但从论文叙事上，4 类方案里有两个类都属于低压分支。  
所以正文建议这样处理：

### 数据结果层

保留 `k = 4` 的细分聚类结果。

### 论文主文层

可合并表述为 3 个宏观类：

1. `票务高压治理型`
2. `服务票务复合型`
3. `低压稳态型`

这样做的好处是：

- 保留数据细分结果
- 正文表达更稳、更清楚
- 不会让读者陷入过细的类间差异

## 7. 类名怎么命名

本轮类名不是黑箱生成，而是根据聚类画像表中各主题的 `priority_index` 和 `dissatisfaction` 水平命名。

命名逻辑是：

- 哪个主题优先级最高
- 哪个主题不满意程度最高
- 总体治理压力高还是低

所以类名本身就是解释结果，而不是单纯标签。

需要特别说明：

- `票务高压治理型` 不是指“只有票务一个因素高压”。
- 它表示该类景区的 `total_priority_index` 处在相对较高水平，同时在三个治理主题中，`票务预约与入园体验` 的 `priority_index` 最高，因此票务链条是主导矛盾。
- 所以更准确的理解是：**整体治理压力较高，且票务问题最突出**。

## 8. 景区原有类别比较怎么做

这里不是聚类，而是按原有 `category_name` 分组，对以下值取组均值：

- `total_priority_index`
- `mean_dissatisfaction`
- 各主题 `priority_index`

由此可以回答：

- 哪类景区整体治理压力更高
- 哪类景区主导问题更偏票务、服务还是交通

## 9. 治理紧迫度分层怎么做

治理紧迫度分层不是聚类，而是辅助排序结果。

做法：

- 先根据 `total_priority_index` 对全部景区排序
- 再按排序结果分成 `高 / 中 / 低` 三档

也就是说：

- 聚类回答“谁和谁的问题结构相似”
- 紧迫度分层回答“谁更该优先治理”

含义：

- `高`：当前最需要优先治理
- `中`：有一定治理压力，但优先级次于高档
- `低`：当前相对稳态，可后续观察

## 10. 可视化方法

### 10.1 聚类散点图

文件：

- `experiment8_cluster_scatter.png`

方法：

- 先对标准化后的聚类特征做 `PCA`
- 降到二维后作图

作用：

- 便于观察景区之间的治理结构相似性

### 10.2 树状图

文件：

- `experiment8_cluster_dendrogram.png`

方法：

- 对同一套标准化特征做层次聚类树状图

作用：

- 便于展示类之间的层次关系
- 对小样本论文展示很有帮助

## 11. 主要输出文件

- `experiment8_cluster_scatter.png`

### 辅助解释结果

- `experiment8_category_compare.csv`

- `experiment8_urgency_tiers.csv`
  治理紧迫度分层结果。

## 12. 论文中如何解释

建议分三段写：

### 12.1 原有类型比较

先回答：

- 哪些类型景区整体治理压力更高
- 哪些类型景区主导问题不同

### 12.2 治理画像聚类

再回答：

- 哪些景区在问题结构上更相似
- 它们相似在票务、服务还是交通

### 12.3 治理紧迫度分层

最后回答：

- 哪些景区当前最值得优先治理

## 13. 注意事项

- 这版聚类完全依赖当前论文版 IPA 指标
- 如果 IPA 主题结构变化，聚类结果也要同步重算
- 当前样本量只有 20 个景区，聚类结果更适合用于解释和归纳，不适合做过强的统计泛化
